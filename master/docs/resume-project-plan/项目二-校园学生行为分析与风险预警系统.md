# 项目二：校园学生行为分析与风险预警系统

## 1. 项目定位

面向校园管理场景，基于门禁日志、借阅日志、网络日志和用户基础信息等多源数据，对学生学习、生活、网络使用和活跃度变化进行处理与分析，构建学生行为画像、群体聚类和风险预警能力。

项目二重点不是“算法有多复杂”，而是突出多源数据处理与分析能力。它承担简历中的真实性和 Java 工程化基础，体现你能完成从数据接入、清洗、特征抽取、聚类画像到预警看板的完整闭环。

项目二不建议强行加入 Agent，也不建议把深度学习写成直接判断学生风险的核心模型。优化后可以把深度学习定位为“行为特征表征增强”：先通过 Python 完成特征矩阵处理，再用 scikit-learn 做标准化和 K-Means 聚类，PyTorch AutoEncoder 只用于学习低维行为表示，Java 侧继续负责业务流程、接口调用、结果写入和异常提示。

这里应重点突出：

1. 多源日志接入与清洗。
2. 用户维度特征融合。
3. 学生行为画像。
4. Python + scikit-learn K-Means 聚类。
5. PyTorch AutoEncoder 表征学习增强。
6. FastAPI 模型服务化与 Java 调用集成。
7. 风险评分与预警同步。
8. Spring Boot 后端工程化能力。

## 2. 建议技术栈

- 后端开发：Java、Spring Boot、Spring MVC、Spring Data JPA、Maven
- 数据库与缓存：MySQL、Redis
- 任务调度：Quartz
- 数据分析：Python、Pandas、NumPy、scikit-learn、K-Means、PyTorch AutoEncoder、规则评分、特征归一化
- 模型服务：FastAPI、Java HttpClient、模型服务异常处理
- 数据源：门禁日志、借阅日志、网络日志、用户基础信息
- 前端展示：HTML、CSS、JavaScript、ECharts

## 3. 数据源设计

项目已有或建议保留的数据源：

| 数据源 | 示例表/文件 | 作用 |
| --- | --- | --- |
| 用户基础信息 | campus_user | 用户 ID、学院、专业、年级等基础维度 |
| 门禁日志 | access_log | 分析教学楼、图书馆、宿舍等空间行为 |
| 借阅日志 | borrow_log | 分析学习资源使用和借阅信用 |
| 网络日志 | network_log | 分析在线时长、访问类别、网络异常 |
| 分析快照 | analysis_data | 存储用户维度的特征和风险分 |
| 风险预警 | risk_warning / warning_log | 存储预警类型、等级、处理状态 |

## 4. 多源数据处理流程

整体链路：

```text
多源数据接入 -> 数据清洗 -> 时间窗口聚合 -> 行为特征抽取 -> 特征融合 -> Python 特征矩阵 -> scikit-learn/PyTorch 聚类分析 -> 用户画像 -> 风险评分 -> 预警同步 -> 看板展示
```

### 4.1 数据接入

- 从 CSV 或 MySQL 中读取门禁、借阅、网络和用户信息。
- 通过 user_id 关联不同数据源。
- 保证日志数据可追溯到具体用户。

### 4.2 数据清洗

- 统一时间格式。
- 处理空值和缺失字段。
- 去除明显重复记录。
- 识别原始日志中的异常标记。
- 对不完整记录进行降权或跳过。

### 4.3 时间窗口聚合

- 以最近 30 天为主要分析窗口。
- 可扩展为周维度、月维度。
- 对每个用户聚合各类行为指标。
- 支持增量同步，避免每次全量重算。

### 4.4 特征抽取

网络维度：

- avgOnlineHours：日均在线时长。
- studyTrafficRatio：学习流量占比。
- networkActivityCount：网络活动次数。
- abnormalTrafficFlag：异常流量标记。

门禁维度：

- libraryAccessCount：图书馆访问次数。
- classroomAccessCount：教学楼访问次数。
- lateReturnCount：晚归次数。
- avgAccessFrequency：日均出入频率。

借阅维度：

- borrowCount：借阅次数。
- avgBorrowDays：平均借阅天数。
- unreturnedCount：未归还数量。

综合维度：

- activeDays：活跃天数。
- riskScore：综合风险分。
- healthScore：行为稳定度分。
- networkRisk：网络维度风险。
- accessRisk：门禁/作息维度风险。
- borrowRisk：借阅维度风险。

### 4.5 特征融合

- 将不同来源的指标按 user_id 合并。
- 存入 analysis_data。
- 形成统一特征向量，用于聚类、画像和预警。

## 5. 行为特征设计

| 数据源 | 可提取特征 | 分析含义 |
| --- | --- | --- |
| 网络日志 | 日均在线时长、学习流量占比、网络活动次数、异常流量标记 | 判断网络使用节律、学习投入和异常访问情况 |
| 门禁日志 | 教学楼访问次数、图书馆访问次数、宿舍晚归次数、日均出入频率 | 判断学习空间活跃度、作息规律和异常进出情况 |
| 借阅日志 | 借阅次数、平均借阅天数、未归还数量、借阅活跃度 | 判断学习资源使用和借阅信用情况 |
| 用户基础信息 | 学号、学院、专业、年级、用户状态 | 支持用户维度关联、筛选和分组统计 |
| 综合分析表 | riskScore、healthScore、activeDays、各类风险分 | 支持画像、看板和预警输出 |

## 6. 分析指标设计

网络行为分析：

- 日均在线时长：反映网络使用强度。
- 学习流量占比：反映学习资源访问倾向。
- 网络活动次数：反映活跃程度。
- 异常流量标记：反映可能的异常访问或流量突变。

门禁行为分析：

- 教学楼访问次数：反映课堂或学习空间活跃度。
- 图书馆访问次数：反映自主学习活跃度。
- 宿舍晚归次数：反映作息异常风险。
- 日均出入频率：反映校园活动规律。

借阅行为分析：

- 借阅次数：反映学习资源使用。
- 平均借阅天数：反映借阅周期。
- 未归还数量：反映借阅信用风险。
- 借阅活跃度：辅助判断学习投入。

综合分析：

- activeDays：反映整体活跃天数。
- healthScore：综合行为稳定度和学习生活规律性。
- riskScore：综合风险分。
- networkRisk、accessRisk、borrowRisk：分维度风险。

## 7. 聚类与画像

优化后使用 Python ML 服务对学生行为特征进行聚类，Java 侧不再保留本地聚类实现，只负责聚合特征、调用模型服务、写入画像结果和处理服务不可用异常。

聚类链路：

```text
Java 聚合 analysis_data 特征
-> FastAPI 接收用户特征列表
-> Pandas / NumPy 构建特征矩阵
-> StandardScaler 标准化
-> PyTorch AutoEncoder 学习低维行为表征，可关闭
-> scikit-learn KMeans 聚类
-> 返回簇标签、簇中心、样本分配和评估指标
-> Java 写入用户画像、聚类概览和看板数据
```

这样写的好处是：算法栈更贴近当前招聘要求，但项目定位仍然是“多源数据处理 + 行为画像 + Java 后端工程化”，不会把项目包装成不真实的深度学习项目。

特征向量建议包括：

```text
avgOnlineHours
studyTrafficRatio
libraryAccessCount
borrowCount
networkActivityCount
classroomAccessCount
lateReturnCount
activeDays
avgAccessFrequency
avgBorrowDays
unreturnedCount
networkRisk
accessRisk
borrowRisk
abnormalTrafficFlag
absenteeFlag
riskScore
healthScore
```

聚类后可以生成以下画像标签：

- 学习活跃型：图书馆/教学楼访问较高，学习流量占比较高，风险较低。
- 作息异常型：晚归次数较高，门禁风险较高。
- 网络偏高型：在线时长较高，娱乐流量占比或异常流量较高。
- 低活跃关注型：activeDays、借阅次数、教学楼访问次数偏低。
- 均衡稳定型：各项指标较均衡，风险分较低。

## 8. 风险评分与预警

风险评分建议采用“规则评分 + 聚类偏移”的组合方式。

规则评分：

- 在线时长过高、学习流量占比过低，增加 networkRisk。
- 教学楼访问偏低、晚归次数过高，增加 accessRisk。
- 未归还数量较高、借阅活跃度下降，增加 borrowRisk。
- activeDays 过低，增加综合风险。

聚类偏移：

- 如果用户所属簇整体风险较高，则增加关注权重。
- 如果用户特征明显偏离稳定群体，则生成群体偏移预警。

预警类型建议：

- 综合风险预警。
- 网络行为异常。
- 行为考勤异常。
- 学习投入异常。
- 借阅信用异常。
- 群体偏移预警。

## 9. 后端模块设计

建议拆分为：

1. DataInitializer：初始化 CSV 数据。
2. DataCleaningUtil：清洗和解析数据。
3. AnalysisComputationService：构建用户分析快照。
4. FeatureExtractorService：抽取用户多维特征。
5. KMeansService：执行聚类并生成画像标签。
6. UserProfileService：生成用户画像。
7. WarningService：同步和处理风险预警。
8. TaskMonitorService：管理任务执行和日志。
9. AnalysisRestController：提供分析结果接口。
10. WarningController：提供预警列表和处理接口。

## 10. 定时任务设计

使用 Quartz 设计任务：

- 数据同步任务：同步新增日志。
- 画像更新任务：重新计算用户画像。
- 聚类分析任务：更新群体分布。
- 预警检查任务：同步风险预警。
- 日志导出任务：导出分析或预警结果。

## 11. 看板设计

建议展示：

- 用户总数、预警数量、高风险用户数量。
- 网络、门禁、借阅三个维度的统计概览。
- 聚类分布图。
- 用户画像详情。
- 风险趋势。
- 预警列表和处理状态。

## 13. 表达边界

项目二应强调“行为分析与风险预警”，不建议写成医学健康评估或心理健康判断。

可以使用：

- 关注风险
- 行为异常
- 学习生活节律变化
- 学习投入变化

- 诊断
- 健康结论

- 医学评估

## 14. 面试解释口径

如果被问“算法部分难度是否足够”，可以这样回答：

项目二的重点是多源数据融合和用户行为画像，算法上以 scikit-learn K-Means 和规则评分为主，并用 PyTorch AutoEncoder 做行为表征增强，更多体现数据处理、特征设计、模型服务化和后端工程化。为了补足时序预测和模型评估能力，我又设计了项目一，专门训练负荷趋势预测、模型评估和异常检测链路。

如果被问“为什么不用深度学习”，可以这样回答：

校园行为项目的数据维度以规则性和可解释性为主，K-Means 和规则评分更容易解释给管理人员，也便于定位具体风险原因。所以我没有把深度学习写成直接预测学生风险的核心模型，而是用 PyTorch AutoEncoder 做特征表征增强，并以 scikit-learn K-Means 作为主聚类方法，通过 FastAPI 提供给 Java 后端调用。

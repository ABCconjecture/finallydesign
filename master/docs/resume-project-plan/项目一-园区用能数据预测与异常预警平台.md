# 项目一：园区用能数据预测与异常预警平台

## 1. 项目定位

面向园区或企业运营场景，基于小时级或日级用能数据，构建用能趋势预测、异常波动识别、风险预警和分析报告生成平台。

该项目技术上贴近电力交易算法岗位中的负荷预测、数据分析、模型上线和效果跟踪，但项目本身不直接声明为电力现货交易系统，避免被追问真实交易规则、出清机制和策略收益。

项目一承担简历中的算法匹配度，重点体现：

1. Python 数据处理能力。
2. 时序特征工程能力。
3. 预测模型训练与评估能力。
4. 异常检测与预警输出能力。
5. Java 后端调用 Python 模型服务的工程化能力。

## 2. 岗位匹配点

| 岗位要求 | 项目对应能力 |
| --- | --- |
| 负荷预测、价格预测基础 | 用能数据预测、时序特征工程、MAE/RMSE/MAPE 评估 |
| 模型优化与算法策略建议 | 模型对比、预测残差分析、异常波动归因 |
| 数据处理与分析能力 | 多字段清洗、时间对齐、缺失值处理、滑动窗口特征 |
| Python 与 Pandas/NumPy | Python 数据分析链路 |
| SQL 与数据维护 | MySQL 入库、预测结果表、预警记录表 |
| 模型上线效果跟踪 | FastAPI 模型接口、Java 后端调用、预测结果可视化 |
| AI 能源应用跟踪 | AI 辅助生成分析摘要，不作为项目主线 |

## 3. 建议技术栈

- 后端开发：Java、Spring Boot、Spring MVC、Spring Data JPA、Maven
- 数据库与缓存：MySQL、Redis
- 任务调度：Quartz
- 数据分析：Python、Pandas、NumPy
- 机器学习：scikit-learn、K-Means、Isolation Forest、RandomForest、XGBoost
- 时序预测：ARIMA、Darts、LSTM 或 PyTorch 基础模型
- 模型服务：FastAPI、REST API
- 可视化：HTML、CSS、JavaScript、ECharts
- AI 辅助分析：Spring AI 或 Spring AI Alibaba，建议只写“了解/基础实践”

## 4. 数据设计

建议先使用公开数据或模拟数据完成实践。数据字段可以设计为：

| 字段 | 含义 |
| --- | --- |
| timestamp | 时间戳，小时级或日级 |
| park_id | 园区或企业编号 |
| load_kwh | 用电量 |
| temperature | 温度 |
| humidity | 湿度，可选 |
| is_workday | 是否工作日 |
| is_holiday | 是否节假日 |
| hour | 小时 |
| weekday | 星期 |
| device_alarm_count | 设备告警数量，可选 |
| production_level | 生产或运营强度，可选 |

如果没有真实数据，可以先生成模拟数据，保证存在以下规律：

1. 白天高、夜间低。
2. 工作日高、周末低。
3. 温度对用能有一定影响。
4. 偶发峰值或骤降作为异常样本。
5. 存在一定随机噪声。

## 5. 实现链路

整体链路：

```text
数据接入 -> 数据清洗 -> 特征工程 -> 预测建模 -> 模型评估 -> 异常检测 -> Java 后端调用 -> 预警与看板展示
```

### 5.1 数据清洗

需要完成：

1. 时间字段解析，统一为标准时间格式。
2. 按小时或日期排序。
3. 缺失值填充，例如均值、中位数、前向填充。
4. 异常值初步处理，例如负数用电量、明显不合理值。
5. 按园区或用户维度分组，保证训练样本连续。

可使用技术：

- Python
- Pandas
- NumPy
- MySQL

### 5.2 特征工程

核心特征建议：

1. 时间特征：hour、weekday、month、is_workday、is_holiday。
2. 周期特征：sin_hour、cos_hour，避免模型误解 23 点和 0 点距离很远。
3. 滞后特征：load_lag_1、load_lag_2、load_lag_24、load_lag_168。
4. 滑动窗口：load_roll_24_mean、load_roll_24_std、load_roll_7d_mean。
5. 外部特征：temperature、humidity、device_alarm_count。
6. 峰谷特征：peak_period、valley_period。

这些特征能支撑面试中解释“为什么预测模型能捕捉用能规律”。

### 5.3 模型选择

建议分三层实现，不要一开始就追求复杂模型：

第一层：基线模型

- 前一天同小时值
- 最近 24 小时均值
- 最近 7 天同小时均值

第二层：机器学习模型

- Linear Regression
- RandomForest
- XGBoost
- LightGBM，可选

第三层：时序或深度学习模型

- ARIMA
- Darts
- LSTM 或 TCN 入门版
- PyTorch 基础模型

简历里重点写“使用基线模型与机器学习模型进行效果对比”，深度学习只在真实完成后再写。

### 5.4 评估指标

至少输出：

- MAE：平均绝对误差。
- RMSE：均方根误差。
- MAPE：平均绝对百分比误差。
- 与基线模型相比的提升比例。

面试解释口径：

MAE 便于理解平均误差规模，RMSE 对大误差更敏感，MAPE 便于表达相对误差。基线模型用于证明模型不是只在训练集上看起来有效。

### 5.5 异常检测

建议用两种方法：

1. 残差阈值法：预测值与真实值差距超过阈值时判定异常。
2. Isolation Forest：基于特征向量识别异常样本。

预警等级可以设计为：

- 低风险：误差超过历史残差 90% 分位。
- 中风险：误差超过历史残差 95% 分位。
- 高风险：误差超过历史残差 99% 分位，或连续多小时异常。

预警字段建议：

| 字段 | 含义 |
| --- | --- |
| warning_id | 预警 ID |
| park_id | 园区编号 |
| warning_time | 预警时间 |
| actual_load | 实际用电量 |
| predicted_load | 预测用电量 |
| residual | 残差 |
| warning_level | 风险等级 |
| warning_reason | 预警原因 |
| status | 处理状态 |

## 6. 模型服务化

推荐架构：

```text
Python 模型服务 FastAPI
        |
        | REST API
        v
Spring Boot 后端
        |
        v
MySQL / Redis / Quartz / ECharts
```

FastAPI 接口建议：

- POST /predict：输入园区 ID 和预测时间范围，返回预测曲线。
- POST /detect-anomaly：输入真实值和预测值，返回异常点。
- GET /model-metrics：返回模型指标。

Java 后端接口建议：

- /api/energy/predictions：查询预测结果。
- /api/energy/warnings：查询预警记录。
- /api/energy/tasks/run：手动触发预测任务。
- /api/energy/dashboard：看板统计数据。

## 7. Java 后端模块

建议拆成：

1. EnergyDataController：数据导入和查询。
2. PredictionController：预测结果查询。
3. EnergyWarningController：预警列表和处理。
4. PredictionTaskService：定时调用 Python 模型服务。
5. EnergyAnalysisService：写入预测结果、计算统计指标。
6. ModelClient：封装 FastAPI 调用。

## 8. 看板展示

页面建议：

- 用能趋势曲线。
- 预测曲线与真实曲线对比。
- 异常点标注。
- MAE/RMSE/MAPE 指标卡片。
- 预警列表。
- 任务执行记录。

## 9. 理论框架验证结果

已使用模拟园区小时级用能数据验证过“特征工程 -> 预测建模 -> 残差异常检测 -> 预警输出”的理论链路。

验证结果摘要：

```text
baseline_mae: 17.309
model_mae: 8.729
model_mape: 5.19%
mae_improvement: 49.57%
anomaly_precision: 0.60
anomaly_recall: 1.00
framework_status: PASS
```

结论：该方案在简历叙述上具备闭环，可以证明你理解特征工程、时序预测、模型评估和异常检测之间的关系。

需要注意：当前验证使用模拟数据，不应在简历中写成真实电力交易数据。真实实践时应补充 Python 数据科学环境，并至少完成一个可运行 Demo。

## 10. 实践顺序

推荐按下面顺序完成：

1. 配置 Python 环境：Pandas、NumPy、scikit-learn、FastAPI。
2. 准备模拟数据或公开数据。
3. 完成数据清洗和特征工程脚本。
4. 实现基线模型和 RandomForest/XGBoost 模型。
5. 输出 MAE、RMSE、MAPE。
6. 实现残差异常检测。
7. 用 FastAPI 封装预测接口。
8. 用 Spring Boot 调用 FastAPI。
9. 写入 MySQL 并展示 ECharts 看板。
10. 最后再考虑加入 Spring AI 辅助分析摘要。

## 11. 简历表达建议

项目名：园区用能数据预测与异常预警平台

项目介绍：

面向园区运营场景设计并实现用能数据预测与异常预警平台，基于历史用能、设备状态、时段、温度等多维数据完成清洗、特征构建、趋势预测和异常波动识别。系统支持预测任务调度、模型服务调用、预测结果入库、预警记录管理和趋势看板展示，用于辅助运营人员发现用能异常和峰谷波动风险。

技术栈：

Java、Spring Boot、Spring MVC、Spring Data JPA、MySQL、Redis、Quartz、Python、Pandas、NumPy、scikit-learn、Darts、FastAPI、ECharts

项目亮点：

1. 构建小时、星期、节假日、滞后特征、滑动均值、滑动标准差等时序特征，支持未来 24 小时或 7 天用能趋势预测。
2. 使用 MAE、RMSE、MAPE 对模型效果进行评估，并与历史同期基线模型进行对比，验证预测效果。
3. 基于预测残差和异常检测模型识别异常波动，生成风险等级和预警记录。
4. 使用 FastAPI 封装 Python 模型服务，Java 后端通过 REST API 调用预测接口，并将结果写入 MySQL。
5. 通过 Quartz 定时触发数据更新、预测任务和预警同步，支持预测曲线、异常点和模型指标可视化。

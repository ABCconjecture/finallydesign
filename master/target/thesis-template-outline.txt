校园网用户行为分析与画像系统设计与实现
▲▲▲▲▲
摘■要
▲▲▲▲▲
随着高校信息化建设的持续推进，校园网络、门禁系统和图书借阅系统在运行过程中不断沉淀大量行为数据。这些数据从不同侧面反映了学生的学习投入、作息规律、网络使用习惯与校园活动状态，也为学业预警、行为识别和管理辅助提供了客观依据。传统依赖人工经验或单一维度统计的管理方式，难以对多源异构数据进行统一处理，也难以对学生行为状态做出连续、客观且可解释的判断。针对上述问题，本文设计并实现了一套基于多源数据融合的校园网用户行为分析与画像系统。
系统采用基于 Java Web 的 B/S 三层架构工程化实现，后端以 Spring Boot、Spring MVC、Spring Data JPA 为核心开发框架，数据库采用 MySQL，任务调度采用 Quartz，分布式会话采用 Spring Session 与 Redis，聚类分析采用 Weka 中的 K-Means 算法。系统以用户基础信息、门禁日志、网络日志和借阅日志为数据来源，围绕数据导入、清洗、特征提取、健康度计算、用户画像生成、风险预警和可视化展示形成完整闭环。在模型设计上，本文构建了由学业投入、作息规律、网络健康和履约信用四个维度组成的健康度评价模型，并将其与风险分、行为规则和群体画像联动，实现了对用户状态的多维分析与异常识别。
基于当前真实样本数据，系统已完成 1000 名用户、约 93.9 万条行为日志的导入与分析，形成 2000 条分析快照、1000 条用户画像，并在真实数据基础上生成多种类型的风险预警结果。测试结果表明，系统能够较稳定地完成数据接入、增量分析、画像聚类、预警管理、权限控制和任务监控等功能，具备较好的完整性、可解释性和扩展性。本文的研究对于高校行为分析系统的轻量级构建和校园管理辅助决策具有一定参考价值。

DESIGN AND IMPLEMENTATION OF A USER BEHAVIOR ANALYSIS AND PROFILING SYSTEM FOR CAMPUS NETWORKS
▲▲▲▲▲
ABSTRACT
▲▲▲▲▲
▲▲▲▲▲
With the continuous development of campus informatization, large amounts of behavioral data have been accumulated in campus networks, access control systems, and library borrowing systems. These data reflect students' learning engagement, daily routine, network usage habits, and campus activities from different perspectives, and therefore provide important support for academic warning, behavior analysis, and management assistance. Traditional management methods based on manual experience or single-dimensional statistics are unable to integrate multi-source heterogeneous data effectively, and they also fail to provide continuous, objective, and explainable judgments on students' behavioral status. To address these issues, this paper designs and implements a campus network user behavior analysis and profiling system based on multi-source data fusion.
The system is implemented using an engineering-oriented B/S architecture based on Java Web technologies. The backend is built with Spring Boot, Spring MVC, and Spring Data JPA, while MySQL is used as the core database. Quartz is adopted for task scheduling, Spring Session with Redis is used for distributed session sharing, and Weka K-Means is used for user clustering and profiling. The system takes user basic information, access control logs, network logs, and borrowing logs as data sources, and builds a complete workflow covering data import, cleansing, feature extraction, health score calculation, user profiling, risk warning, and visualization. In the model design, a health score evaluation model is constructed from four dimensions, including academic engagement, routine regularity, network health, and credit performance. The health score is further linked with risk score, rule-based warning logic, and cluster labels to realize multi-dimensional analysis and anomaly detection.
Based on the current real sample data, the system has completed the analysis of 1,000 users and approximately 939,000 behavioral logs, producing 2,000 analysis snapshots, 1,000 user profiles, and multiple categories of warning records. Experimental results show that the system can stably support data ingestion, incremental analysis, profiling, warning management, permission control, and task monitoring. Therefore, the proposed system provides a feasible and practical solution for lightweight campus behavior analysis and management support.

KEYWORDS:●campus network;●multi-source data fusion;●user profiling;●health score;●risk warning

1●绪论
1.1●研究背景
在数字校园建设不断深入的背景下，高校管理活动已经从传统的线下记录逐步转向数据驱动模式。网络认证、门禁通行、图书借阅、线上学习和校园应用访问等系统在日常运行中持续生成结构化或半结构化数据。对于学校而言，这些数据不仅是业务运行痕迹，更是理解学生行为模式和辅助管理决策的重要基础。特别是在学业指导、沉迷识别、异常行为发现和校园资源优化等方面，行为数据分析具有较高的应用价值。
然而，校园行为数据天然具有来源分散、结构不一致、时间粒度不同和业务语义复杂等特点。例如，网络日志更偏向于在线行为和流量特征，门禁数据更能反映出入规律和活动区域，借阅记录则从学习投入角度提供补充信息。如果仅从单一数据源出发，容易对学生状态做出片面判断。现实中，依赖人工经验或简单阈值的分析方式，已经难以满足高校精细化管理的需求。
从管理实践出发，学校更加关心的并不是某一条具体日志，而是“这个用户当前状态如何”“是否存在持续性风险”“为什么系统将其标记为重点关注对象”“接下来应采取什么措施”。这就要求行为分析系统不仅能够完成数据统计，还应具备多源融合、连续分析、用户分群、异常预警和可解释展示的能力。特别是在本科毕业设计场景下，系统既要具备完整的业务闭环，又要兼顾实现难度、数据可获得性和工程可落地性。
基于此，本文围绕校园网用户行为分析场景，设计并实现了一套基于多源数据融合的校园网用户行为分析与画像系统。该系统以真实样本数据为基础，以用户为分析核心，综合使用健康度模型、聚类分析、预警规则和可视化界面，力图实现“多源数据统一接入—行为特征建模—画像构建—异常预警—管理辅助”的完整分析链路。
1.2●国内外研究现状
国内针对校园网用户行为分析与用户画像的研究，近年来主要集中在以下几个方向：多源异构数据融合、校园行为建模、聚类分群、异常预警和系统平台实现。已有研究普遍指出，仅依赖单一网络日志难以完整刻画学生状态，必须将网络行为、门禁记录、图书借阅、认证记录等多类数据综合起来，才能更加客观地反映用户行为特征。
在多源融合与画像应用方向，汪钊的研究强调了校园网用户画像应建立在多源数据融合基础上，通过标签体系构建和画像模型设计提升用户刻画的完整性；葛丽静的研究则更重视从系统设计与实现角度打通数据采集、分析和应用展示的全过程；陈伟宁针对校园网络行为日志开展用户画像研究，说明了行为标签、画像生成和展示平台之间的关系。这些研究为本文在“多源融合 + 画像展示”方面提供了直接借鉴。
在异常行为分析方面，严帆等研究指出，校园网异常行为不应仅根据某个简单指标进行判断，而应综合考虑超长在线、异常流量、共享账号、作息异常等多个行为特征。该思路与近年来安全领域常用的 UEBA 思想较为一致，即通过用户与实体行为特征识别偏离正常模式的异常对象。对于校园场景而言，这种多规则联动的思路更符合实际需求，因为学生行为具有明显的时间性和情境性，单一分数往往难以说明问题。
在系统实现技术方面，国内部分研究选择 Hadoop、Spark 等大数据平台处理更大规模的数据，强调海量日志分析能力；也有不少研究采用 Java Web、关系数据库和轻量级分析组件构建校园行为分析系统，以保证项目可控性和可落地性。考虑到本科毕设的实现边界，后者更符合当前课题的实际条件。相比之下，国外相关研究在算法层面更倾向于图模型、时序建模、深度学习和用户建模技术，应用领域覆盖推荐系统、网络安全、教育数据挖掘和在线学习分析等方向。国外研究在算法精度和模型复杂度上更具优势，但对数据质量、算力资源和工程条件的要求也更高。
综合国内外研究现状可以看出，当前研究已经基本形成了三个共识：第一，校园行为分析必须建立在多源数据融合基础上；第二，画像分群不仅要有算法结果，还要有可解释标签和实际应用场景；第三，异常预警应从单一阈值判断逐步走向多规则联动和动态更新。本文的系统设计正是在这些研究思路的基础上展开的，并结合本科毕业设计场景，在系统完整性、工程实现能力和结果可解释性之间进行了平衡……

2●系统总设计
2.1●设计任务
1. 建立多源数据接入与管理能力。系统应能够统一处理用户基础信息、网络日志、门禁日志和借阅日志，为行为分析提供可靠数据源。
2. 建立面向用户的行为分析能力。系统应能够围绕用户形成统计指标、风险分和健康度结果，而不是停留在原始日志层面。
3. 建立用户画像能力。系统应支持将用户划分为若干具有代表性的群体，并对群体特征做出可解释描述。
4. 建立异常预警能力。系统应基于多规则联动识别潜在风险对象，并支持管理员查看、处理和复核。
5. 建立可视化与管理辅助能力。系统应通过首页、分析页、画像页、预警页等界面帮助管理人员理解当前状态并快速找到重点对象。
6. 建立工程支撑能力。系统应支持权限控制、定时调度、任务日志、操作审计和基础性能优化，保证系统具备完整性和可维护性。

除功能任务外，系统还需要满足一些非功能需求。首先，系统应保证数据处理与页面展示口径一致，避免首页、分析页和预警页之间出现互相矛盾的结果。其次，系统应具备一定的可解释性，让管理人员明白“结果是怎么得来的”。再次，系统应具备基本的扩展性，便于未来接入新的数据源或新的分析方法。最后，系统在当前数据规模下应保持较稳定的响应速度和可用性。


2.2●设计原理
本系统遵循“多源融合、统一建模、分层处理、结果可解释、实现可落地”的设计原理。
2.2.1 多源融合原则
系统设计不以单一日志为分析基础，而是以用户为核心，将网络、门禁和借阅行为统一映射到同一用户实体下。这样既可以从不同角度观察用户行为，也能避免仅依赖某一类数据造成的偏差。
2.2.2 统一建模原则
在分析口径上，系统不允许不同页面使用不同的指标定义。健康度、风险分、群体标签和预警状态都应基于同一份分析快照和同一套规则生成，从而保证首页、用户列表、多维分析和预警管理页面之间结果一致。
2.2.3 分层处理原则
系统采用典型的分层架构，前端负责展示和交互，控制器负责请求转发和响应包装，服务层负责业务处理和模型计算，数据层负责持久化存储。通过清晰的层次划分，可以降低模块耦合度，提高系统的可维护性。
2.2.4 结果可解释原则
对于校园管理系统而言，仅有分数和图表是不够的。系统必须能够回答“为什么健康度低”“为什么被划入某一群体”“为什么触发预警”等问题。因此在模型和页面设计上，系统始终将可解释性放在重要位置，通过四维健康度拆解、群体命名、规则来源展示和异常时间线等方式增强可理解性。
2.2.5 实现可落地原则
考虑到本科毕业设计在时间、资源和工程复杂度上的限制，本文不追求过于复杂的大数据平台和深度模型，而是在真实样本数据基础上选择技术成熟、实现可控且表达清晰的方案，以确保系统能够稳定实现并顺利答辩。
2.3 系统总体架构设计
系统采用基于 Java Web 的 B/S 三层架构工程化实现，整体结构可分为表现层、业务层、数据层和调度支撑层。
2.3.1 表现层设计
表现层由静态页面、原生 JavaScript 和 ECharts 图表组件组成，主要负责首页、多维分析、用户列表、预警管理、用户画像、任务状态和账号管理等页面的交互展示。页面通过统一的 API 封装与后端通信，图表部分采用折线图、雷达图、环形图和条形图等方式展示趋势、分布和对比信息。为了提高非技术用户的使用体验，页面在后续优化中加入了颜色分层、分页组件、空状态提示、重点关注模块和统一的视觉样式。
2.3.2 业务层设计
业务层基于 Spring Boot 构建，由控制器层和服务层共同组成。控制器负责接收前端请求、参数绑定和统一返回结果，服务层负责系统核心逻辑。业务层中的主要服务包括：
1.数据清洗与初始化服务：负责导入和清洗样本数据；
2.分析计算服务：负责分析快照、健康度和风险分的计算；
3.聚类画像服务：负责特征构建、K-Means 聚类和群体标签生成；
4.预警服务：负责规则判断、预警生成、关闭和处理；
5.登录认证与权限服务：负责登录、角色权限和分布式会话；
6.调度与监控服务：负责定时任务、执行日志和操作审计。
这种服务拆分方式使系统既能保持业务清晰，又便于后续模块化扩展。
2.3.3 数据层设计
数据层以 MySQL 为核心业务数据库，通过 Spring Data JPA 进行访问。系统采用实体类与 Repository 相结合的方式完成业务表映射和查询。数据层不仅存储用户和日志数据，还用于保存分析结果、画像结果、预警记录、任务状态和系统用户等信息。数据层是整个系统的事实来源，因此后续页面展示、聚类计算和预警逻辑均以数据库结果为准。
2.3.4 调度与支撑层设计
系统采用 Quartz 完成定时任务调度，并使用 JDBC JobStore 持久化调度信息。这样既能支持任务自动执行，也便于在多节点部署时实现任务共享和状态同步。Redis 则主要负责 Spring Session 的分布式会话存储，为后台多实例部署提供支持。调度支撑层使系统不仅是一个“查询系统”，同时也具备持续更新和运维管理的能力。
2.3.5 系统业务流程设计
系统整体业务流程可概括为：
1. 导入并清洗原始样本数据；
2. 将不同来源日志按用户进行聚合；
3. 提取行为特征并生成分析快照；
4. 基于分析快照计算健康度和风险分；
5. 基于分析结果进行用户聚类并生成画像标签；
6. 基于规则和画像结果生成风险预警；
7. 将分析、画像和预警结果在前端页面进行可视化展示；
8. 通过 Quartz 对分析、画像和预警进行增量更新，并在任务看板中跟踪运行状态。
这一流程保证了系统从数据输入到结果输出的完整性，也使后续论文测试和答辩讲解更具逻辑性。
2.4 功能模块设计
结合系统总体目标，本文将系统划分为以下核心功能模块。
2.4.1 数据管理模块
数据管理模块负责样本数据的导入、清洗、转换和初始化。该模块通过工具类和初始化流程完成 CSV 数据读取、字段转换和数据库写入，并在系统首次运行时生成基础分析快照和初始预警记录。
2.4.2 首页看板模块
首页看板模块主要展示系统总体运行状态和关键管理指标，包括用户总数、平均健康度、高风险用户数、开放预警数、趋势变化、低健康重点关注对象、群体健康对比和任务状态摘要等内容。该模块强调“入口价值”，要求管理者在首页即可获得当前系统状态的总体判断。
2.4.3 多维分析模块
多维分析模块面向单个用户展开深度分析，支持按学号或用户标识进行查询。该模块整合用户当前画像、历史分析、开放预警、健康度拆解、异常行为时间线和与群体均值差异等内容，旨在帮助管理者从结果进一步追踪原因。
2.4.4 用户管理模块
用户管理模块主要承担用户检索、筛选、分页查看和快捷跳转功能。用户列表不仅展示基础信息和健康度，还通过颜色分层和低健康提醒帮助管理者快速定位重点对象，并支持从列表直接跳转到分析详情、预警记录和群体详情。
2.4.5 预警管理模块
预警管理模块集中展示预警记录和风险对象，支持预警统计、等级分布、规则排行、高风险用户分页、未处理预警分页、低健康待复核和人工处理等功能。该模块是系统将分析结果转化为管理动作的核心入口。
2.4.6 用户画像模块
用户画像模块围绕聚类结果开展展示，包括群体分布、群体名称、群体平均雷达图、风险来源、典型样本和簇内分页用户详情。该模块是系统体现“画像能力”和“群体理解能力”的主要页面。
2.4.7 任务调度与运维模块
任务调度与运维模块负责分析、画像和预警任务的自动执行、日志记录、失败重试、暂停恢复和操作审计。通过该模块，系统具备一定的可观测性和运维能力，也使其从“静态分析系统”提升为“可持续运行系统”。
2.4.8 账号与权限模块
账号与权限模块负责后台登录、角色控制、账号启停、密码修改和用户信息管理。通过区分管理员与查看员角色，可以更好地保护敏感操作和管理功能，提升系统安全性。
2.5 数据库设计
系统数据库采用 MySQL 8.0，字符集为 utf8mb4，存储引擎为 InnoDB。数据库既需要承担基础业务数据存储，也要支持任务调度、预警管理和页面查询，因此其设计必须兼顾规范性和性能。
2.5.1 核心业务表设计
系统的核心业务表包括：
`campus_user`：保存用户基础属性信息；
`access_log`：保存门禁日志；
`network_log`：保存网络行为日志；
`borrow_log`：保存借阅日志；
`analysis_data`：保存分析快照结果；
`user_profile`：保存用户当前画像；
`risk_warning`：保存预警记录；
`system_user`：保存系统后台用户信息；
`task_execution_state`、`task_execution_log`、`task_operation_audit`：保存任务状态、执行日志和审计记录。
这些表围绕用户维度建立了从原始日志到派生结果的完整数据链路，为后续分析和展示奠定基础。
2.5.2 表关系设计思路
虽然系统当前主要通过应用层维护数据一致性，但逻辑上各表关系是明确的：`access_log`、`network_log`、`borrow_log`、`analysis_data`、`user_profile` 和 `risk_warning` 均围绕 `campus_user.user_id` 建立关联。系统采用“原始日志表 + 派生结果表 + 管理支撑表”的结构，使原始数据与分析结果相互分离，有利于后续扩展。
2.5.3 数据库优化设计
在系统优化阶段，数据库针对当前分析场景完成了以下增强：
1. 为 `campus_user.student_id` 增加唯一约束，保证学号唯一性；
2. 为 `user_profile.user_id` 增加唯一约束，保证一人一份当前画像；
3. 为 `access_log`、`network_log`、`borrow_log` 增加 `user_id + 时间字段` 组合索引，提升单用户分析与增量分析效率；
4. 为 `risk_warning` 增加状态、等级和时间组合索引，提升预警管理和分页查询效率；
5. 为 `analysis_data` 增加时间相关索引，提升分析历史查询和趋势统计效率；
6. 为用户筛选相关字段增加辅助索引，提升搜索与筛选性能。
这些优化使数据库从“可以运行”进一步提升到“更适合分析场景”，也增强了系统的工程完整性。

3.1●开发环境与关键技术
本系统后端采用 Java 17 作为开发语言。Java 语言具有面向对象特性强、生态成熟、适合企业级开发等优势，能够较好地支撑本课题中涉及的业务逻辑、数据库访问、任务调度和接口开发。
Spring Boot 是本系统的核心开发框架。相比传统的 Servlet 或手工搭建 Spring 工程的方式，Spring Boot 通过自动配置和依赖整合显著降低了项目搭建难度。系统通过 Spring MVC 构建 RESTful 接口，通过 Spring Data JPA 进行数据库访问，通过依赖注入降低模块之间耦合，从而使系统结构更清晰，开发效率更高。
MySQL 作为系统核心业务数据库，负责存储用户信息、原始日志、分析结果、用户画像、预警记录、任务状态和账号信息。由于当前系统数据类型以结构化数据为主，且数据规模处于中等水平，因此 MySQL 能够很好地满足业务要求。Redis 主要用于 Spring Session 分布式会话存储，使后台登录状态在多节点部署下保持一致。Quartz 则用于构建分析、画像和预警三类核心任务的定时调度机制，并通过 JDBC JobStore 支持任务持久化和集群共享。
在数据分析部分，系统采用 Weka 的 SimpleKMeans 实现用户聚类。相比自行实现算法或引入更重的机器学习平台，Weka 更适合当前本科项目的实现边界。一方面，它可以直接与 Java 工程集成；另一方面，其聚类结果便于与后端业务逻辑联动。前端展示则主要依赖 ECharts 完成各类统计图表渲染，HTML、CSS 和 JavaScript 负责页面结构、样式和交互行为。
3.2●多源数据采集与清洗设计
3.2.1 数据来源设计
系统当前采用脱敏样本数据进行离线导入，数据类型包括用户基础信息、门禁日志、网络日志和借阅日志四类。其中，用户信息主要用于描述分析主体的静态属性；门禁日志反映用户在图书馆、教学楼和宿舍等空间中的活动行为；网络日志反映在线行为、流量规模和网络类别；借阅日志则提供学习参与与履约情况的补充信息。
这种数据源组合能够较好地覆盖校园行为分析所需的核心维度，同时避免对真实业务系统的直接接入依赖，更适合毕业设计场景中的研究与实现。
3.2.2 数据清洗流程设计
数据清洗流程是系统可靠运行的前提。本文系统在数据导入时主要执行以下步骤：
1. 文件读取与字段映射：将 CSV 文件中的列映射到对应实体字段；
2. 空值清理：对关键字段为空的数据做过滤或填补；
3. 时间转换：将字符串时间统一转换为日期或时间对象；
4. 数值格式统一：将时长、流量、次数等字段转换为统一数值类型；
5. 分类标准统一：对网络类别、地点类型等字段进行标准化；
6. 用户关联：确保日志记录中的用户标识与基础用户表一致；
7. 数据入库：将清洗后的数据写入对应业务表。
通过这一流程，系统能够较好地处理样本数据中的格式差异和缺失问题，使后续分析计算更稳定。
3.2.3 分析窗口设计
在实际分析中，系统并非简单读取用户全部历史日志，而是围绕最新有效时间构建分析窗口。当前实现采用近 30 天的行为窗口作为主要分析区间，从而使分析结果更能反映用户近期状态。网络日志、门禁日志和借阅日志都会按窗口过滤后再参与分析，这种做法有利于避免历史极端值对当前状态判断造成过强干扰。
3.2.4 数据质量控制设计
为了避免数据质量问题传递到分析层，系统在清洗和计算阶段还加入了以下质量控制策略：
- 对缺失严重的记录不参与关键特征计算；
- 对明显不合理的时长或流量值进行兼容处理；
- 对无法关联到有效用户的数据不参与后续分析；
- 对布尔型异常标识统一转化为 0/1 结构；
- 对结果计算中的空列表和空时间进行默认保护。
这些处理保证了系统在样本数据不完全理想的情况下仍能较稳定地运行。
3.3 健康度计算模型设计
3.3.1 健康度模型设计思路
传统校园行为分析系统往往直接将若干风险指标累加成总风险，再通过“100 减风险”得到一个简单健康度。这种做法容易导致结果区分度不足，也难以向管理者解释“得分为什么高或低”。因此，本文重新构建了基于多维行为语义的健康度模型，使健康度既能体现用户状态，也便于前端拆解展示。
健康度模型将用户状态划分为四个维度：学业投入、作息规律、网络健康和履约信用。四个维度分别从学习行为、校园活动规律、网络使用结构和借阅行为规范性等角度刻画用户状态。这种设计兼顾了校园场景的业务意义和模型可解释性。
3.3.2 特征选择与权重设置
系统采用以下权重设置：
- 学业投入：35%
- 作息规律：25%
- 网络健康：25%
- 履约信用：15%
其中，学业投入维度权重最高，是因为校园行为分析的最终目标之一是辅助识别学习状态；作息规律和网络健康对异常行为识别具有重要意义，因此占据相同权重；履约信用维度主要用于体现借阅规范性，权重相对较低，但仍保留对整体状态的补充价值。
具体到指标层面，系统使用了学习流量占比、课堂门禁次数、图书馆门禁次数、借阅次数、晚归次数、平均门禁频率、日均在线时长、异常流量标识、未归还数量和平均借阅时长等指标共同参与计算。
3.3.3 模型计算方法
系统先从分析窗口中提取原始行为特征，再根据指标的正向或逆向含义分别做标准化处理。一般而言，学习流量占比、课堂参与、图书馆参与和借阅次数属于正向指标，值越大越好；晚归次数、异常流量、未归还数量等属于逆向指标，值越大风险越高。
健康度计算公式如下：
`HealthScore = 0.35 × Academic + 0.25 × Routine + 0.25 × Network + 0.15 × Credit`
为了统一系统口径，风险分定义为：
`RiskScore = 100 - HealthScore`
这种设计能够使健康度和风险分在首页、用户列表、分析页和预警页中保持一致，减少解释冲突。
3.3.4 分层显示策略
为便于页面展示和管理判断，系统进一步将健康度划分为三个区间：
- 健康度 < 60：低健康度，重点关注对象；
- 60 ≤ 健康度 < 75：中等健康度，需要持续观察；
- 健康度 ≥ 75：整体状态较好。
这一分层结果已应用于用户列表、分析页提醒卡、首页重点关注和画像页面群体统计中，形成一致的视觉表达和业务判断标准。
3.4 用户画像与 K-Means 聚类设计
3.4.1 聚类目标
用户画像的目标并不是简单给用户贴标签，而是通过行为特征相似性将用户划分为若干行为模式相近的群体，再结合群体特征解释用户可能的整体状态。这样既能帮助管理人员从个体走向群体理解，也能为预警联动和风险识别提供辅助依据。
3.4.2 聚类特征构建
系统从最新分析快照中抽取用户聚类特征，主要包括日均在线时长、学习流量占比、图书馆门禁次数、课堂门禁次数、借阅次数、网络活跃次数、晚归次数、活跃天数、平均门禁频率、平均借阅时长、未归还数、网络风险、门禁风险、借阅风险、异常流量标识、考勤异常标识、风险分和健康度等。
这些特征涵盖了学习行为、校园活动、网络结构和综合状态等多个维度，能够较全面地反映用户近期行为模式。
3.4.3 聚类过程设计
系统在执行聚类前先完成特征行构建和标准化处理，再将样本输入 Weka 的 SimpleKMeans。聚类数默认设置为 5，这是综合当前样本规模、可解释性和页面展示效果后确定的结果。若实际特征模式过于单一，系统也具备退化为单簇的保护逻辑，以避免算法异常。
聚类完成后，系统会计算各簇的中心特征，并基于簇中心与全体样本的相对位置自动生成群体名称与标签摘要。相比写死名称的方式，这种方式更能贴合真实数据分布。
3.4.4 群体命名与解释设计
当前真实数据下，系统形成了以下五个群体：
- 高投入学习型：学习投入高、图书馆或课堂参与积极；
- 在线风险型：在线时长显著较高，网络风险特征明显；
- 校园参与型：校园活动参与较为常规，属于样本中占比最高的普通群体；
- 稳健自律型：整体健康度高、行为较规律、风险较低；
- 夜间失衡型：作息规律偏弱，晚归与夜间行为问题相对明显。
在页面设计中，系统不仅展示群体名称，还进一步展示群体平均雷达图、群体风险来源和典型样本，使“画像”真正具备解释意义。
3.5 风险预警规则设计
3.5.1 预警规则设计思路
预警模块采用“规则驱动 + 画像联动”的设计方式。系统不直接依据单一得分触发预警，而是根据不同风险场景设计多类规则，并将用户当前群体特征作为辅助判断依据。这样可以降低误判率，也使预警结果更容易解释。
3.5.2 预警类型设计
当前系统主要包含以下类型：
1. 网络行为异常：如在线时长过高、异常流量命中、网络行为结构失衡；
2. 行为考勤异常：如课堂参与低、门禁规律失衡、晚归次数较多；
3. 学业投入异常：如学习流量占比过低、图书馆参与不足、借阅投入偏低；
4. 群体偏移预警：用户当前行为明显偏离所属群体均值；
5. 画像群体风险：用户所属群体本身具备较高风险倾向；
6. 综合风险：多项风险叠加导致需要重点关注。
3.5.3 预警等级与状态管理
系统对预警等级进行分类处理，主要包括 `CRITICAL`、`HIGH`、`MEDIUM` 和 `LOW`。与此同时，预警状态被划分为开放状态与已处理状态，管理员可以对开放预警进行人工处理，并记录处理人、处理时间和处理备注。
3.5.4 低健康待复核机制
考虑到有些用户可能健康度较低，但暂时未达到正式预警规则触发条件，系统增加了“低健康未预警待复核”模块。该模块不是正式预警，而是一种辅助提醒，用于帮助管理人员发现系统规则之外仍值得关注的对象，从而形成更完整的管理闭环。

4●系统实现
本4.1 登录与权限管理实现
系统后台实现了完整的账号与权限管理功能。账号信息存储在 `system_user` 表中，用户登录后通过 Session 保存身份信息，系统根据角色决定可访问的操作范围。其中，`ADMIN` 角色拥有任务控制、预警处理、账号管理等敏感权限；`VIEWER` 角色主要用于浏览数据，不允许执行敏感变更。
从实现角度看，系统通过认证服务完成登录校验、密码加密存储和会话写入，通过拦截器对页面访问和接口访问进行统一权限判断。对于需要管理员权限的接口，如预警处理、任务暂停、任务重试和账号管理等，系统在后端统一校验角色，而不是依赖前端限制按钮显示，从而保证权限边界更加可靠。
在多节点部署场景下，系统使用 Spring Session + Redis 保存后台登录态，避免单机 Session 导致的登录状态不一致问题。该设计也为系统后续的扩展部署提供了基础条件。
4.2 首页看板实现
首页是系统最重要的综合入口，其实现目标不是简单展示若干图表，而是帮助用户在最短时间内把握系统全局状态。首页实现主要包括以下几个部分。
第一，顶部指标卡展示用户总数、平均健康度、高风险用户数和开放预警数，并结合上一次真实分析快照展示变化趋势。为了避免图表信息过载，“较上一分析周期”的变化不再放在趋势图中，而是直接通过红、绿、灰三色放在顶部指标卡上，下降显示红色，上升显示绿色，持平显示灰色。
第二，趋势图采用单折线切换模式。用户可以在“平均健康线”和“高风险用户数线”之间切换，从而根据需要观察系统总体状态的变化。由于系统当前真实分析日期数量有限，前端在文案上明确采用“与上一分析快照比较”的表述，避免出现伪造连续七天趋势的问题。
第三，首页新增了低健康度重点关注模块，用于展示当前健康度低于阈值的重点对象。该模块帮助管理人员在不进入用户列表的情况下快速锁定高关注用户。
第四，首页新增了群体健康对比和昨日变化摘要。群体健康对比从人数、平均健康度、低健康人数和预警覆盖率等维度展示不同群体差异；昨日变化摘要则以更简洁的方式表达平均健康度、高风险人数和低健康人数相对上一分析周期的变化情况。
第五，首页还保留了最近任务状态摘要，使管理者能够快速知道分析、画像和预警任务是否执行正常。这一设计增强了首页的运营和管理价值。
4.3 多维分析功能实现
多维分析页面是系统解释能力最强的页面之一，主要服务于对单个用户的深入研判。页面支持按学号进行搜索，并在查询成功后联动展示分析结果、历史记录、用户画像、开放预警和行为洞察。
在页面结构上，系统首先展示用户的基础信息、健康度、风险分、当前群体和预警状态，使使用者能够快速形成整体印象。随后，通过雷达图呈现用户在多个行为维度上的综合表现。为了进一步增强解释能力，系统还实现了以下四个增强模块：
1. 较上一分析周期变化：展示健康度、风险分、日均在线、晚归次数等相对上一次分析的变化；
2. 健康度四维拆解：将综合健康度拆解为学业投入、作息规律、网络健康和履约信用四部分；
3. 异常行为时间线：将晚归、异常流量、借阅异常、开放预警等事件按逻辑顺序组织展示；
4. 用户与群体均值差异：对比当前用户与所属群体平均水平的差距。
此外，当用户健康度低于阈值时，页面顶部会出现健康度过低提醒卡，并给出简要说明和建议关注方向。这种实现方式使多维分析页面真正承担起“从结果到原因”的解释任务，而不只是展示一组指标。
4.4 用户列表与用户管理实现
用户列表页面面向全量用户，强调检索效率、重点识别和快捷跳转能力。页面实现了按学号、姓名、学院、专业、群体和状态等维度的筛选，并支持分页浏览。为了增强直观性，系统对健康度采用三级颜色策略：低于 60 分显示红色，60 至 75 分显示橙色，高于 75 分显示绿色。
在交互设计上，页面对低健康用户增加了提醒标识，并在每条记录后提供分析详情、预警记录和群体详情入口，使管理者可以从列表快速进入下一步分析。相比只有表格信息的传统用户管理页面，这种实现方式更强调“先发现问题，再深入分析”的工作流。
在技术实现上，用户列表的数据来源不再依赖前端拼接，而是由后端统一返回用户基础信息、群体名称、健康度、风险状态和提醒标识等结果，从而保证列表与首页、分析页、画像页的数据口径一致。
4.5 预警管理实现
预警管理页面承担系统风险发现与处理的主要职责。页面在原有未处理预警列表和高风险用户列表基础上，进一步增强为包含统计、分布、排行和处理四类能力的管理页面。
首先，页面实现了预警处置漏斗，用于从累计预警、当前开放预警、已处理预警、涉及预警用户和重点风险用户五个阶段观察系统预警运行情况。该模块更适合答辩和实际管理者从全局角度理解系统状态。
其次，页面实现了预警等级分布与规则命中排行。预警等级分布按照 `CRITICAL`、`HIGH`、`MEDIUM`、`LOW` 统计当前开放预警的等级占比；规则命中排行则展示最常命中的规则类型，便于理解当前主要风险来自网络行为、学业投入还是作息失衡等场景。
再次，页面强化了低健康未预警待复核模块。该模块用于补充正式预警体系之外的弱风险对象，帮助管理者发现“尚未触发硬性规则但值得关注”的用户。虽然当前真实数据下该列表可能为空，但空状态本身也是合理结果，说明系统并未伪造数据。
最后，页面实现了管理员处理预警的完整闭环。处理操作会记录当前真实登录用户、处理备注和处理时间，并从未处理列表中移除对应项。这样既保证了操作审计，又增强了预警管理页面的业务闭环能力。
4.6 用户画像与任务调度实现
用户画像页面围绕聚类结果展开实现。页面首先展示当前五类群体的分布情况，随后针对选中群体展示群体平均雷达图、主要风险来源、典型用户样本和簇内分页用户详情。典型用户样本包括最稳健样本、最高风险样本和最接近群体中心样本，这种设计有利于解释群体特征并增强答辩展示效果。
为了避免一次性渲染大量簇内用户导致页面信息拥挤，系统将簇内用户详情改为后端分页接口，前端通过分页组件逐页渲染。这一改动使画像页面在用户规模增大时仍能保持良好的可读性和交互体验。
在任务调度方面，系统主要设计了三类自动任务：增量分析任务、画像更新任务和预警检查任务。三类任务按固定时序执行，确保系统分析、画像和预警结果能够持续更新。与此同时，系统还实现了任务状态看板，展示任务最新执行时间、执行结果、失败日志、重试记录和操作审计。管理员可以在页面中手动执行、暂停、恢复或重试任务，并对相关操作进行审计跟踪。
通过将任务调度能力和任务可视化能力结合，系统不仅能够“分析一次”，还能够“持续分析”，这也是当前系统与单纯毕设演示页面的重要区别之一。

5.3.2 健康度分布分析
最新分析结果表明，健康度模型已经具备较好的区分能力。当前健康度分布为：
- 健康度小于 60 分的用户约 100 人；
- 健康度位于 60 到 75 分之间的用户约 624 人；
- 健康度大于等于 75 分的用户约 276 人。
这意味着健康度模型已经基本避免了“全部用户得分集中在高分段”的问题，能够将低健康对象、中间状态对象和整体表现较好的对象区分开来。与页面中的红、橙、绿三色显示策略结合后，管理者能够更加快速地识别重点用户。
5.3.3 用户画像分析
当前聚类结果形成五个可解释群体：
- 高投入学习型：202 人；
- 在线风险型：10 人；
- 校园参与型：553 人；
- 稳健自律型：145 人；
- 夜间失衡型：90 人。
从人数分布来看，校园参与型占比最高，说明多数用户行为表现处于较常规状态；高投入学习型与稳健自律型体现了相对积极和稳定的行为模式；夜间失衡型和在线风险型虽然人数不多，但在风险识别中价值更高，更适合作为重点关注对象。由此可见，系统聚类结果不仅具有统计意义，也具备较强的管理解释意义。
5.3.4 预警结果分析
预警结果显示，系统能够根据不同规则生成多种类型的风险提醒，并在页面中进一步拆分为等级分布、规则排行和重点风险对象。相比单纯依赖总风险分排序的方式，当前系统通过规则来源、群体标签和低健康度提示共同作用，使预警结果更具可解释性，也更贴近校园管理实际场景。
从预警处置角度看，系统已经形成“发现—展示—处理—审计”的完整链路，管理员可以基于预警页面和多维分析页面开展进一步研判，这说明系统已经初步具备管理辅助能力。
5.4 系统不足与优化方向
虽然本文系统已经具备完整的业务闭环和较好的演示效果，但从研究与工程两个角度看，仍存在一定不足。
首先，当前真实分析快照日期仍然较少。虽然系统已经将趋势图改为基于真实分析快照的展示方式，避免了伪造连续趋势的问题，但如果想进一步提升趋势分析效果，仍需持续积累按天生成的分析数据。
其次，当前系统数据来源以脱敏样本导入为主，尚未直接对接真实校园认证系统、门禁系统或图书系统接口。因此，系统在实际工程形态上更偏研究型和验证型平台，而非面向生产环境的全实时系统。未来若面向真实部署，还需进一步完善接口接入、数据同步和异常恢复机制。
再次，当前聚类算法采用 K-Means，虽然在可实现性和可解释性方面具有优势，但在处理更复杂的时序关系、用户关系或高维异构行为模式时仍有局限。未来可以考虑引入图模型、时间序列建模或更复杂的异常检测方法。
最后，虽然数据库已经完成第一阶段优化，包括唯一约束和关键索引补充，但若未来日志量继续增长，仍可以考虑按时间分区、日志归档、中间统计表和专用分析引擎等更深层次优化手段。
结■论
▲▲▲▲▲
本文围绕校园网用户行为分析与画像需求，设计并实现了一套基于多源数据融合的校园网用户行为分析与画像系统。系统以用户基础信息、网络日志、门禁日志和借阅日志为数据基础，完成了从数据导入、清洗、特征提取、健康度计算、K-Means 聚类画像、多规则风险预警，到可视化展示、任务调度和权限控制的完整系统实现。
在系统设计方面，本文采用基于 Spring Boot 的 Java Web B/S 三层架构，并结合 MySQL、Redis、Quartz、Weka 和 ECharts 等关键技术，构建出兼顾功能完整性、实现可控性和工程表达力的系统方案。在模型设计方面，本文提出了由学业投入、作息规律、网络健康和履约信用组成的四维健康度评价模型，并通过统一风险分口径、群体聚类和多规则预警设计，增强了系统的可解释性。在页面展示方面，系统围绕首页、多维分析、预警管理和用户画像等模块进行了持续优化，使系统从“可运行原型”逐步提升为“接近最终提交形态的毕设系统”。
从当前真实数据运行结果来看，系统已能够在 1000 名用户、约 93.9 万条行为日志基础上稳定运行，并形成用户画像、风险预警和管理看板等完整结果。

毕业论文模板版式和特殊符号的说明

（1）毕业论文(设计）采用A4幅面打印，靠左边胶装。
（2）全文版面页边距：上3cm，下3cm，左3cm，右2.6cm，装订线：0.0cm。
（3）摘要、正文、致谢、参考文献和目录的【内容】均设置成【两端对齐】。
（4）每一章另起一页，总结和致谢不编入正文。
（5）凡是文中出现“■”代表1个空格【全角】；“●”代表1个空格【半角】；
“▲▲▲▲▲”表示在此处按照【宋体小四号字体，段前段后均为0行、1.15倍行距的设置空一行】。
目录页由于“■”占有一定大小，故看起来行间距大于1.15，但实际间距仍为1.15（点开“断落选项卡”）。
封面页仅供参考，如有变化，按学校的制定为准。
